WHY THE TESTS ARE BROKEN (Not The Code)
================================================================================

Problem Pattern:

  ❌ BROKEN TEST:
  ────────────────────────────────────────────────────────────────────────────
  var mockAdapter = new Mock<IProtocolAdapter>();
  mockConnectionPool.Setup(p => p.ReleaseAsync(...)).Callback(...);

  // Uses reflection to bypass client API
  poolField?.SetValue(client, mockConnectionPool.Object);

  // Calls mock directly, never enters client try-catch
  await mockAdapter.Object.ConnectAsync();  // ← WRONG

  Assert.True(releaseWasCalled);  // Tests mock, not client
  ────────────────────────────────────────────────────────────────────────────

  What Should Happen:
  ────────────────────────────────────────────────────────────────────────────
  // Call through real client API
  var client = new SurrealDbClient(options);
  await Assert.ThrowsAsync<ConnectionException>(() => client.ConnectAsync());

  // Verify real cleanup: pool statistics, not mock expectations
  var stats = client.GetPoolStatistics();
  Assert.Equal(stats.AvailableConnections, stats.TotalConnections);
  ────────────────────────────────────────────────────────────────────────────

Test Design Anti-Patterns Found:
  1. Reflection-based field injection (bypasses encapsulation)
  2. Mock object testing instead of real behavior
  3. Brittle Moq verification (fragile to changes)
  4. Incomplete client code path coverage

TESTING PRINCIPLES TO ENFORCE
================================================================================

✅ DO:
  • Test public behavior, not mocks
  • Use mocks only for external dependencies
  • Write integration tests for client scenarios
  • Keep unit tests fast (<100ms each)
  • Document why tests matter

❌ DON'T:
  • Use reflection to bypass encapsulation
  • Test mock objects instead of real code
  • Inject internal state via reflection
  • Verify implementation details
  • Mix concerns in a single test

Pattern: See ARCHITECTURE_RECOMMENDATIONS.md for detailed principles

COMPARISON: GOOD vs BAD TEST DESIGN
================================================================================

InjectionVulnerabilityTests (GOOD DESIGN - 71 tests, 100% pass):
────────────────────────────────────────────────────────────────
[Fact]
public void Vuln3_EscapeIdentifier_WithDangerousCharacters_Throws()
{
    // Uses reflection for access only (private static method)
    var method = typeof(SurrealDbClient).GetMethod("EscapeIdentifier",
        BindingFlags.NonPublic | BindingFlags.Static);

    // Tests REAL behavior: dangerous chars throw
    var exception = Assert.Throws<TargetInvocationException>(() =>
        method!.Invoke(null, new object[] { "id;DROP TABLE" }));

    // Verifies real outcome
    Assert.IsType<ValidationException>(exception.InnerException);
}

Results: Clear, maintainable, tests real code path ✅

ResourceManagementTests (BAD DESIGN - 9 tests, 78% pass):
────────────────────────────────────────────────────────
[Fact]
public async Task F2_ConnectAsync_WhenConnectAsyncThrows_ReleasesConnection()
{
    // Reflection abuse to bypass client API
    var poolField = typeof(SurrealDbClient)
        .GetField("_connectionPool", BindingFlags.NonPublic | BindingFlags.Instance);
    poolField?.SetValue(client, mockConnectionPool.Object);

    // Calls mock directly, never through client
    await mockAdapter.Object.ConnectAsync();

Results: Brittle, misleading, tests mock not code ❌
