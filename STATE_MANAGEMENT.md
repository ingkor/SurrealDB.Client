## Overview

State management in SurrealDB.Client follows the **Unit of Work pattern** with automatic change tracking, similar to Entity Framework Core. The `ISurrealDbSession` provides:

- **Single-threaded state container**: Manages a coherent set of changes
- **Automatic change detection**: Snapshots and compares entity state
- **Transaction scope**: All session changes committed atomically
- **Memory-efficient updates**: Only modified properties sent to server

### Core Principle

> A session is a **unit of work**: a bounded context where all changes are tracked, compared against snapshots, and committed atomically.

## Entity States

Every tracked entity has one of five states:

### 1. **Detached** (0)

Entity not tracked by any session.

```csharp
// Detached - no session
var user = new User { Id = "user:1", Name = "John" };
```

**Characteristics:**
- Not associated with session
- Changes ignored
- Cannot be saved directly
- Must be attached before SaveChangesAsync()

**Transitions:**
```
Detached --[Add/Update/Remove]--> {Added, Modified, Deleted}
```

### 2. **Added** (4)

New entity added to session for insertion.

```csharp
using var session = client.CreateSession();

var newUser = new User { Name = "Jane", Email = "jane@example.com" };
session.Add(newUser);  // State: Added

await session.SaveChangesAsync();
// Executes: INSERT INTO users VALUES (...)
// State: Unchanged (if successful)
```

**Characteristics:**
- No snapshot (new entity)
- Next SaveChangesAsync() generates INSERT
- Primary key may be generated by server
- All properties included in insert

**Lifetime:**
```
Added (after session.Add)
  ↓ SaveChangesAsync()
Unchanged (with server-generated ID)
```

### 3. **Unchanged** (1)

Entity retrieved from database with no modifications.

```csharp
using var session = client.CreateSession();

var user = await session.FindAsync<User>("user:1");
// State: Unchanged (snapshot created)
```

**Characteristics:**
- Snapshot captured from database
- All properties match snapshot
- SaveChangesAsync() is no-op for this entity
- Can transition to Modified or Deleted

**Transitions:**
```
Unchanged --[Modify property]--> Modified
Unchanged --[session.Remove()]--> Deleted
Unchanged --[session.Detach()]--> Detached
```

### 4. **Modified** (3)

Entity loaded and then modified.

```csharp
using var session = client.CreateSession();

var user = await session.FindAsync<User>("user:1");
// State: Unchanged

user.Email = "newemail@example.com";  // Property changed
// State: Modified (detected automatically)

await session.SaveChangesAsync();
// Executes: UPDATE users:1 SET Email = 'newemail@example.com'
// Only changed properties sent
```

**Characteristics:**
- Has snapshot (original values)
- Only modified properties in UPDATE
- Change detection compares property-by-property
- SaveChangesAsync() generates differential UPDATE

**Snapshots vs. Current**:
```csharp
var entry = session.ChangeTracker.Entry(user);
entry.GetOriginalValue("Email");  // "oldemail@example.com"
user.Email;                         // "newemail@example.com"
entry.GetModifiedProperties();     // ["Email"]
```

**Transitions**:
```
Modified --[SaveChangesAsync()]--> Unchanged
Modified --[session.Reload()]--> Unchanged (discard changes)
Modified --[session.Remove()]--> Deleted
```

### 5. **Deleted** (2)

Entity marked for deletion.

```csharp
using var session = client.CreateSession();

var user = await session.FindAsync<User>("user:1");
session.Remove(user);  // State: Deleted

await session.SaveChangesAsync();
// Executes: DELETE FROM users WHERE id = 'user:1'
```

**Characteristics:**
- Marked for deletion, not actually deleted yet
- Previous state discarded
- SaveChangesAsync() generates DELETE
- After delete, entity is Detached

**Transitions**:
```
Deleted --[SaveChangesAsync()]--> Detached (deleted successfully)
Deleted --[session.Reload()]--> Unchanged (cancel deletion)
```

### State Transition Diagram

```
                ┌─────────────────┐
                │   Detached      │
                │   (NEW)         │
                └────────┬────────┘
                         │
        ┌────────────────┼────────────────┐

        ▼                ▼                ▼

```

## Session Lifecycle

### Creation & Disposal

```csharp
// Session is typically short-lived
using var session = client.CreateSession();
// Session created, ChangeTracker initialized

// ... perform operations ...

await session.SaveChangesAsync();
// Changes persisted, ChangeTracker cleared

// Session disposed - connections returned to pool
// (Automatic via using statement)
```

### Session as Boundary

```
REQUEST SCOPE
├─ Create Session
├─ Query: SELECT ... (snapshot created)
├─ Modify: entity.Property = value (tracked)
├─ Add: session.Add(new entity)
├─ Remove: session.Remove(entity)
├─ SaveChangesAsync()
│  └─ Transaction boundary
│  └─ All changes atomic
│  └─ ChangeTracker cleared
└─ Dispose Session
  └─ Connections to pool
  └─ Memory cleaned up
```

### Multiple Sessions

```csharp
// Session 1 - independent scope
using (var session1 = client.CreateSession())
{
    var user = await session1.FindAsync<User>("user:1");
    user.Email = "new1@example.com";
    await session1.SaveChangesAsync();
}

// Session 2 - fresh scope
using (var session2 = client.CreateSession())
{
    var user = await session2.FindAsync<User>("user:1");
    // State: Unchanged
    // Sees latest value from session1's save

    user.Email = "new2@example.com";
    await session2.SaveChangesAsync();
}

// Sessions are independent
// No state sharing between them
```

## Change Tracking Mechanism

### Snapshot Strategy

When an entity is loaded or added, a **snapshot** is created:

```
LOAD: SELECT * FROM users:1
  └─ Create: Snapshot { Email: "old@example.com", Name: "John" }
  └─ State: Unchanged

MODIFY: user.Email = "new@example.com"
  └─ Compare: snapshot.Email vs. user.Email
  └─ Detect: Property changed
  └─ State: Modified

SAVE: SaveChangesAsync()
  └─ Generate: UPDATE users:1 SET Email = 'new@example.com'
  └─ Send only changed properties
  └─ Update snapshot
  └─ State: Unchanged
```

### Property-Level Granularity

```csharp
var user = await session.FindAsync<User>("user:1");
// Snapshot: { Id: "user:1", Name: "John", Email: "old@test.com", Age: 30 }

user.Email = "new@test.com";      // Property changed
user.UpdatedAt = DateTime.UtcNow;  // Property changed

// Query only changed properties
var entry = session.ChangeTracker.Entry(user);
var changed = entry.GetModifiedProperties();  // ["Email", "UpdatedAt"]

await session.SaveChangesAsync();
// Generated SurrealQL:
//   UPDATE users:1 SET Email = 'new@test.com', UpdatedAt = <timestamp>
// NOT: UPDATE users:1 SET { ... all properties ... }
```

### Snapshot Comparison Algorithm

```csharp
public bool IsPropertyChanged<T>(T entity, string propertyName)
{
    var snapshot = GetSnapshot(entity);
    var current = entity.GetType()
        .GetProperty(propertyName)
        .GetValue(entity);

    // Use ValueComparer for complex types
    return !AreEqual(snapshot[propertyName], current);
}

// Complex type comparison
private bool AreEqual(object a, object b)
{
    if (ReferenceEquals(a, b)) return true;
    if (a == null || b == null) return false;

    // Use custom ValueComparer if configured
    if (HasValueComparer(a.GetType()))
        return GetValueComparer(a.GetType()).Compare(a, b);

    // Default: Equality
    return a.Equals(b);
}
```

## ChangeTracker API

### Entry & State Access

```csharp
using var session = client.CreateSession();
var user = await session.FindAsync<User>("user:1");

// Get entry wrapper
var entry = session.ChangeTracker.Entry(user);

// Inspect state
var state = entry.State;  // EntityState.Unchanged
var properties = entry.Properties;  // PropertyEntry[]
var isModified = entry.State == EntityState.Modified;

// Original values
var originalEmail = entry.GetOriginalValue("Email");
var currentEmail = user.Email;

// Modified properties
var modified = entry.GetModifiedProperties();  // string[]
var hasChanges = modified.Any();

// All tracked entities
var allEntries = session.ChangeTracker.TrackedEntities;
var allModified = allEntries.Where(e => e.State == EntityState.Modified);
```

### Property Entry

```csharp
public class PropertyEntry
{
    public string Name { get; }
    public object CurrentValue { get; set; }
    public object OriginalValue { get; }
    public bool IsModified { get; set; }
    public Type Type { get; }
}

// Usage
var emailProperty = entry.Properties.First(p => p.Name == "Email");
Console.WriteLine($"Current: {emailProperty.CurrentValue}");
Console.WriteLine($"Original: {emailProperty.OriginalValue}");
Console.WriteLine($"Changed: {emailProperty.IsModified}");
```

### Reloading & Discarding

```csharp
// Discard changes and reload from database
user.Email = "unsavedvalue@example.com";
await session.Entry(user).ReloadAsync();
// user.Email is now server value again

// Or reload specific property
await session.Entry(user).Property("Email").ReloadAsync();

// Manual state reset
session.Entry(user).State = EntityState.Unchanged;
// Discards all changes for this entity
```

## Usage Patterns

### Pattern 1: Query & Modify

```csharp
using var session = client.CreateSession();

// Load entity (Unchanged)
var user = await session.FindAsync<User>("user:1");

// Modify (Modified)
user.Email = "newemail@example.com";

// Save all changes atomically (Unchanged)
await session.SaveChangesAsync();
// Only Email property sent to server
```

### Pattern 2: Create & Relationships

```csharp
using var session = client.CreateSession();

// Create parent (Added)
var user = new User { Name = "John", Email = "john@example.com" };
session.Add(user);

// Create children (Added)
var order1 = new Order { UserId = user.Id, Amount = 100 };
var order2 = new Order { UserId = user.Id, Amount = 200 };
session.Add(order1);
session.Add(order2);

// Save everything atomically
await session.SaveChangesAsync();
// Three INSERTs in single transaction
```

### Pattern 3: Bulk Updates

```csharp
using var session = client.CreateSession();

// Load multiple entities
var users = await session.Set<User>()
    .Where(u => u.Status == "inactive")
    .ToListAsync();

// Modify all
foreach (var user in users)
{
    user.Status = "active";
    user.UpdatedAt = DateTime.UtcNow;
}

// Save all changes efficiently
await session.SaveChangesAsync();
// Multiple UPDATEs in single transaction
// Only Status and UpdatedAt properties sent per record
```

### Pattern 4: Conflict Handling

```csharp
using var session = client.CreateSession();

try
{
    var user = await session.FindAsync<User>("user:1");
    user.Email = "new@example.com";
    await session.SaveChangesAsync();
}
catch (ConcurrencyException ex)
{
    // Reload current values
    await session.Entry(user).ReloadAsync();

    // Merge changes with server version
    user.Email = "new@example.com";  // Re-apply our change

    // Try again
    await session.SaveChangesAsync();
}
```

### Pattern 5: Cascading Operations

```csharp
using var session = client.CreateSession();

var user = await session.FindAsync<User>("user:1");

// Delete user and all orders (cascade)
session.Remove(user);

// All related orders also marked deleted (if cascade configured)
var orders = await session.Set<Order>()
    .Where(o => o.UserId == user.Id)
    .ToListAsync();

foreach (var order in orders)
{
    session.Remove(order);
}

// Single atomic delete transaction
await session.SaveChangesAsync();
```

## Performance Implications

### Bandwidth Efficiency

**Without change tracking** (naive):
```csharp
var user = await client.GetAsync<User>("user:1");
user.Email = "new@example.com";
await client.UpdateAsync("user:1", user);
// Sends entire User object (all 20+ properties)
```

**With change tracking** (smart):
```csharp
var user = await session.FindAsync<User>("user:1");
user.Email = "new@example.com";
await session.SaveChangesAsync();
// Sends only: { Email: "new@example.com" }
// 10-50x less bandwidth for large objects
```

### Memory Overhead

Snapshots add memory cost:

```
Per Entity: ~1.5x object size
Example: User with 10 properties (1KB) → 1.5KB in memory
With 1000 tracked entities: ~500KB additional

Mitigation:
- Use session.ChangeTracker.DetachMany() to untrack
- Short session lifetimes
- Avoid tracking large datasets
```

### Network Round Trips

**Multi-operation scenario**:
```csharp
// Without session:
await client.UpdateAsync("user:1", user);      // Round trip 1
await client.UpdateAsync("order:1", order);    // Round trip 2
await client.UpdateAsync("order:2", order);    // Round trip 3
// 3 network calls, each waits for previous

// With session:
session.Update(user);
session.Update(order1);
session.Update(order2);
await session.SaveChangesAsync();
// 1 network call, all changes atomic
```

### Change Detection Performance

- **Simple properties** (string, int): O(1) comparison
- **Complex types** (nested objects): Custom ValueComparer needed
- **Collections** (List<T>): Reference comparison (requires manual .ToList())

Avoid in tight loops:
```csharp
// ❌ Inefficient: Change detection per property access
for (int i = 0; i < 1000000; i++)
{
    user.Email = "email" + i;  // Property setter triggers state check
}

// ✅ Efficient: Update once
user.Email = "final@example.com";
```

## Testing Strategies

### Unit Testing: Mock Session

```csharp
public class MockSession : ISurrealDbSession
{
    private Dictionary<string, object> _store = new();

    public async Task SaveChangesAsync()
    {
        // Simulate persistence
        await Task.Delay(10);
    }

    public IQueryable<T> Set<T>(string table) where T : class
    {
        return _store
            .Where(kv => /* filter by type */)
            .Select(kv => kv.Value as T)
            .AsQueryable();
    }

    public async Task<T> FindAsync<T>(string id) where T : class
    {
        _store.TryGetValue(id, out var entity);
        return entity as T;
    }
}

// Usage in tests
var mockSession = new MockSession();
var service = new UserService(mockSession);
var result = await service.GetUserAsync("user:1");
// No database required
```

### Integration Testing: Real Session

```csharp
[Collection("Database")]
public class UserRepositoryTests : IAsyncLifetime
{
    private ISurrealDbSession _session;
    private SurrealDbClient _client;

    public async Task InitializeAsync()
    {
        _client = new SurrealDbClient("ws://localhost:8000");
        _session = _client.CreateSession();
    }

    [Fact]
    public async Task Update_WithSinglePropertyChange_SendsOnlyChangedProperty()
    {
        // Arrange
        var user = new User { Id = "user:1", Name = "John", Email = "old@test.com" };
        _session.Add(user);
        await _session.SaveChangesAsync();

        // Clear for next test
        user = await _session.FindAsync<User>("user:1");

        // Act
        user.Email = "new@test.com";
        var entry = _session.ChangeTracker.Entry(user);
        var modifiedProperties = entry.GetModifiedProperties();

        // Assert
        Assert.Single(modifiedProperties);
        Assert.Equal("Email", modifiedProperties.First());
    }

    public async Task DisposeAsync()
    {
        await _session.DisposeAsync();
        await _client.DisconnectAsync();
    }
}
```

### Snapshot Testing

```csharp
[Fact]
public void Snapshot_CapturesOriginalValues()
{
    // Arrange
    var user = new User { Email = "old@test.com" };
    var entry = session.ChangeTracker.Entry(user);
    var snapshot = entry.GetOriginalValue("Email");

    // Act
    user.Email = "new@test.com";

    // Assert

    Assert.Equal("new@test.com", user.Email);
```

## Best Practices

1. **Keep sessions short-lived** - One request = one session
2. **Don't track large datasets** - Query only what you need
3. **Detach unused entities** - Free memory in long-running processes
4. **Test with real database** - Snapshots are complex to mock accurately
5. **Monitor memory usage** - Snapshots multiply with large objects
6. **Handle concurrency explicitly** - Catch and retry on conflicts
7. **Use ValueComparers for complex types** - Otherwise reference comparison fails
8. **Batch operations** - Multiple changes in single SaveChangesAsync()
